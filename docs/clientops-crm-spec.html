Overview
Przegląd systemu i moduły

ClientOps to wielomodułowy system CRM klasy SaaS przeznaczony do zarządzania klientami, projektami i działaniami marketingowymi (SEO). System oferuje dwa interfejsy: panel administracyjny dla zespołu oraz dedykowane panele klienckie dostępne pod indywidualnymi subdomenami.

👥
CRM — Leady i Klienci
Zarządzanie lejkiem sprzedaży, kontaktami, historią komunikacji. Kanban dla leadów z drag & drop. Segmentacja klientów i tagowanie.
📁
Projekty
Tworzenie i zarządzanie projektami powiązanymi z klientami. Zadania, kamienie milowe, śledzenie postępu. Powiązanie z SLA.
🔍
SEO z AI
Audyty techniczne, analiza słów kluczowych, semantyka treści, integracja z Google Search Console i Analytics 4. Silnik AI.
🎫
SLA i Tickety
System zgłoszeń dla klientów z SLA (czas odpowiedzi / rozwiązania). Automatyczne przekazywanie ticketów do Kanban admina.
📄
Dokumenty
Edytor online (TipTap), wersjonowanie, eksport do PDF. Dokumenty powiązane z projektami, klientami lub ticketami.
🔔
Powiadomienia
Trzy kanały: Email, SMS (SMSAPI.pl), in-app real-time (Reverb). Preferencje per użytkownik z quiet hours dla SMS.
🏢
Panel Klienta
Dedykowana subdomena {slug}.clientops.pl per klient. Podgląd projektów, zgłaszanie ticketów, dostęp do dokumentów i raportów SEO.
📋
Kanban
Ujednolicony Kanban obsługujący leady, tickety i zadania projektowe. Jeden model z polymorphic relations, drag & drop (dnd-kit).

Backend

Technologia Wersja Opis & Uzasadnienie
Laravel 12.x Główny framework PHP do budowy aplikacji webowych. Dostarcza routing, ORM (Eloquent), system kolejek, broadcasting, autoryzację i wiele więcej. Laravel 12 wprowadza ulepszone starter kity, streamlined middleware API oraz lepszą integrację z nowoczesnymi narzędziami frontendowymi.
Największy ekosystem PHP, najlepsza dokumentacja, aktywna społeczność. Dostarcza out-of-the-box wszystko czego wymaga CRM tej skali — bez składania frameworka z osobnych bibliotek.
PHP 8.3+ Nowoczesna wersja PHP z Fiber (asynchroniczność), named arguments, readonly properties i natywne typy wyliczeniowe (enums). Wersja 8.3 wprowadza typed class constants, readonly w klasach anonimowych oraz lepszą obsługę błędów. Wydajność PHP 8.x jest o ~30% wyższa niż PHP 7.4 dzięki JIT compiler.
Wymagany przez Laravel 12. PHP 8.3 to najnowsza stabilna wersja z długim wsparciem (LTS do 2027). Enums i typed properties kluczowe dla modelowania statusów (LeadStatus, TicketPriority).
PostgreSQL 16+ Zaawansowana relacyjna baza danych open-source z pełnym wsparciem dla ACID, JSON/JSONB, full-text search i rozszerzeń (TimescaleDB). PostgreSQL 16 wprowadza równoległe wykonywanie zapytań, lepszy query planner i wsparcie dla logicznej replikacji. Obsługuje zarówno strukturalne dane CRM jak i półstrukturalne dane audytów SEO (JSONB).
MySQL nie obsługuje TimescaleDB ani zaawansowanego JSONB — oba kluczowe dla modułu SEO. PostgreSQL jest standardem dla nowych projektów SaaS wymagających złożonych zapytań analitycznych.
Redis 7.x Szybki in-memory data store działający jako cache, broker kolejek i magazyn sesji. Używany przez Laravel Horizon do dystrybucji jobów, przez Laravel do cache'owania zapytań i przez Reverb do przechowywania stanu połączeń WebSocket. Redis 7.x dodaje Redis Functions i ACL v2 dla lepszej kontroli dostępu.
Jeden komponent pełni trzy role (cache + queue + session) — upraszcza infrastrukturę. Alternatywy (database queue driver) nie skalują się przy obciążeniu jakie generują audyty SEO i powiadomienia.
Laravel Queues — Wbudowany system kolejek Laravel umożliwiający asynchroniczne wykonywanie zadań w tle — bez blokowania odpowiedzi HTTP. Joby mogą być opóźniane, ponawiane przy błędach, priorytetyzowane i rozdzielane na dedykowane kolejki (np. osobna kolejka dla AI, osobna dla emaili). Fundament dla wszystkich operacji długotrwałych w systemie.
Audyt SEO trwa minuty, generowanie PDF sekundy, wysyłka SMS może mieć opóźnienia API — bez kolejek użytkownik czekałby na odpowiedź HTTP. Queues sprawiają że każda akcja jest natychmiastowa z perspektywy użytkownika.
Laravel Sanctum — Lekki system autentykacji dla SPA i API oparty o cookies (stateful) lub tokeny (stateless). Dla panelu admina i panelu klienta używa autentykacji cookie-based — bezpieczniejszej niż tokeny w localStorage, odpornej na XSS. Dla potencjalnych integracji zewnętrznych obsługuje Personal Access Tokens.
Passport to overhead dla SPA (OAuth2 server). Sanctum jest dedykowany dla tego wzorca użycia: SPA na tej samej domenie + opcjonalne API tokeny. Natywna integracja z subdomenami przez SESSION_DOMAIN.
Laravel Policies / Gates — Wbudowany w Laravel system autoryzacji na poziomie zasobów (Policies) i akcji (Gates). Policy to klasa PHP powiązana z modelem Eloquent — definiuje kto może wykonać daną akcję na konkretnym rekordzie (np. czy ten user może edytować ten projekt). Gates to proste domknięcia dla uprawnień niezwiązanych z modelem.
Spatie Permission zarządza rolami — Policies zarządzają dostępem do konkretnych rekordów. To dwa poziomy autoryzacji: "czy masz rolę manager" (Spatie) oraz "czy jesteś przypisany do tego projektu" (Policy). Oba są konieczne.
Spatie Permission — Pakiet implementujący RBAC (Role-Based Access Control) dla Laravel z obsługą ról i uprawnień per guard. Pozwala przypisywać role użytkownikom, definiować granularne uprawnienia (np. tickets.respond, seo.view) i sprawdzać je przez Blade directives i Gate facades. Integruje się z Laravel's Policy system dla ochrony zasobów.
Pisanie własnego RBAC to tygodnie pracy i źródło błędów bezpieczeństwa. Spatie to standard de facto w ekosystemie Laravel — przetestowany, dobrze udokumentowany, z obsługą wielu guardów (admin vs client panel).
Laravel Notifications — Wbudowany w Laravel system powiadomień obsługujący wiele kanałów przez jeden interfejs: email, SMS, database (in-app), broadcast (WebSocket), Slack i inne. Każde powiadomienie to klasa PHP z metodami per kanał — jeden kod obsługuje wszystkie kanały. Preferencje użytkownika decydują które kanały są aktywne.
Alternatywa — osobny kod dla email, osobny dla SMS, osobny dla WebSocket — prowadzi do duplikacji i niespójności. Notifications unifikuje wszystko w jednym miejscu. Kluczowe dla systemu z trzema kanałami (email + SMS + in-app).
Laravel Horizon — Dashboard i supervisor dla kolejek Redis w Laravel, umożliwiający monitoring jobów w czasie rzeczywistym. Automatycznie balansuje workery między kolejkami na podstawie obciążenia, powiadamia o nieudanych jobach i przechowuje statystyki przepustowości. Krytyczny dla heavy asynchronicznych operacji: audyty SEO, generowanie PDF, wysyłka powiadomień.
Bez Horizon kolejki są "czarną skrzynką" — nie wiesz ile jobów czeka, które się nie powiodły, ile trwają. Horizon to observability dla kolejek, niezbędna gdy audyt SEO składa się z 6 różnych jobów uruchamianych seryjnie.
Laravel Reverb — Natywny serwer WebSocket dla Laravel, wprowadzony jako alternatywa dla Pusher bez zewnętrznych zależności. Obsługuje prywatne i publiczne kanały, presence channels (kto jest online) oraz skalowanie przez Redis pub/sub. Używany do in-app powiadomień real-time, aktualizacji statusu ticketów i live Kanban updates.
Pusher kosztuje przy skali i uzależnia od zewnętrznego dostawcy. Reverb działa self-hosted, jest oficjalnie wspierany przez Laravel team i nie wymaga żadnych zmian w kodzie aplikacji względem Pushera (ten sam protokół).
S3 / MinIO — Object storage do przechowywania plików: dokumenty, załączniki, eksportowane PDF, dane SEO. S3 (AWS) lub kompatybilny self-hosted MinIO — identyczne API, wymienne bez zmiany kodu. Laravel Filesystem abstrahuje storage przez ten sam interfejs niezależnie od wybranego drivera. Spatie Media Library korzysta z tego storage dla wszystkich mediów.
Przechowywanie plików na dysku serwera nie skaluje się i komplikuje deployment. S3/MinIO to standard — pliki są poza aplikacją, dostępne z CDN, łatwe do backupu. MinIO na start (self-hosted, zero kosztów), migracja do S3 bez zmiany kodu gdy zajdzie potrzeba.
Spatie Activity Log — Pakiet do rejestrowania każdej akcji w systemie — zarówno adminów jak i klientów. Zapisuje kto (causer), co (subject), kiedy i jaka była zmiana (diff old/new w JSON). Tabela activity_log jest wspólna dla całego systemu, logowane zdarzenia: auth (login/logout), CRM (create/update/delete/convert), Kanban (zmiana statusu), dokumenty, tickety, powiadomienia, role.
Bez audit logu nie wiadomo kto usunął kontakt, kto zmienił SLA policy, kiedy klient po raz pierwszy zalogował się do panelu. Spatie Activity Log jest de facto standardem w Laravel — działa przez trait na modelu, zero custom kodu dla podstawowych operacji CRUD.
Spatie Media Library — Pakiet do zarządzania plikami i mediami powiązanymi z modelami Eloquent, z obsługą konwersji i optymalizacji obrazów. Abstrahuje storage — pliki mogą być przechowywane lokalnie, na S3 lub innym dysku bez zmiany kodu aplikacji. Używany dla załączników dokumentów, plików SEO i eksportowanych PDF.
Bez Media Library każdy upload to ręczne zarządzanie ścieżkami, konwersjami, czyszczeniem. Pakiet dostarcza polymorphic relacje (każdy model może mieć media), automatyczne konwersje obrazów i responsywne URL z jedną linią kodu.
Laravel Browsershot — Wrapper Laravel dla biblioteki Puppeteer (Node.js), umożliwiający generowanie PDF i screenshotów przez headless Chromium. Zapewnia najwyższą jakość renderowania CSS — dokładnie to co widzi przeglądarka, przeniesione do PDF. Generowanie odbywa się asynchronicznie przez kolejkę, użytkownik dostaje powiadomienie gdy PDF jest gotowy.
DomPDF nie obsługuje flexbox/grid — dokumenty TipTap używają nowoczesnego CSS. Browsershot renderuje identycznie jak Chrome, co gwarantuje że eksportowany PDF wygląda tak samo jak podgląd w edytorze.
Laravel Scout + Meilisearch — Scout to abstrakcja full-text search dla Eloquent modeli, Meilisearch to szybki open-source silnik wyszukiwania z relevance ranking i typo-tolerance. Razem umożliwiają błyskawiczne wyszukiwanie w dokumentach, danych SEO i historii klientów bez obciążania PostgreSQL. Meilisearch można hostować self-hosted lub używać Meilisearch Cloud.
PostgreSQL LIKE query nie skaluje się na tysiącach dokumentów i nie obsługuje literówek ani relevance rankingu. Meilisearch zwraca wyniki w <50ms z typo-tolerance — kluczowe dla wyszukiwarki w panelu klienta i admina.
TimescaleDB — Rozszerzenie PostgreSQL optymalizujące przechowywanie i zapytania dla danych time-series (szeregi czasowe). Automatycznie partycjonuje tabele po czasie (hypertables), co przyspiesza zapytania z filtrem po dacie nawet 100x. Idealne dla rankingów słów kluczowych SEO, gdzie zbieramy dziesiątki tysięcy punktów danych dziennie per klient.
Tabela rankingów bez TimescaleDB przy 100 klientach × 500 słów kluczowych × 365 dni = 18 mln wierszy rocznie — standardowe zapytania stają się wolne. TimescaleDB rozwiązuje to bez zmiany stacku, bo to rozszerzenie PostgreSQL.

Frontend

Technologia Opis & Uzasadnienie
React 19 + TypeScript Biblioteka JavaScript do budowy interfejsów użytkownika oparta o komponenty i jednokierunkowy przepływ danych. React 19 wprowadza Server Components, Actions i ulepszony concurrent rendering dla płynniejszego UX. TypeScript dodaje statyczne typowanie, eliminując całą klasę błędów runtime i znacząco przyspieszając development w dużym projekcie.
Największy ekosystem komponentów i narzędzi. TypeScript obowiązkowy przy CRM tej skali — typy na API responses, formularze i modele eliminują całą klasę błędów jeszcze przed uruchomieniem.
Vite Nowoczesny build tool oparty o natywne ES modules i esbuild, zapewniający błyskawiczny cold start serwera deweloperskiego (poniżej 300ms) i Hot Module Replacement bez przeładowania strony. Vite optymalizuje bundle produkcyjny przez Rollup z tree-shaking i code splitting per route. Zastępuje Webpack oferując wielokrotnie szybszy development experience.
Webpack przy projekcie tej wielkości startuje 30–60 sekund. Vite startuje poniżej sekundy. Przy kilkuset komponentach różnica w DX jest znacząca każdego dnia pracy.
Tailwind CSS v4 Utility-first CSS framework gdzie style definiuje się bezpośrednio w klasach HTML, eliminując przełączanie się między plikami CSS i JSX. Tailwind v4 to kompletne przepisanie silnika — oparty o Lightning CSS (Rust), do 10x szybszy build. Konfiguracja wyłącznie przez @tailwindcss/vite plugin, bez pliku konfiguracyjnego JS.
Fundament design systemu — phoenix-design-system definiuje tokeny przez @theme Tailwinda v4. Utility-first eliminuje problem nazewnictwa klas CSS i dead code w dużym projekcie.
phoenix-design-system Własny design system przepisany z ClientOps Bootstrap na Tailwind v4. Zawiera tokeny (@theme: kolory, typografia, spacing, shadows, border-radius, dark mode przez CSS variables) oraz gotowe komponenty CSS (buttons, badges, cards, forms, modal, dropdown, kanban, timeline, wizard i inne). Jeden import (@import "./phoenix-design-system/index.css") daje pełny system.
Zastępuje shadcn/ui — mamy już wszystkie komponenty ze spójnym wyglądem dopasowanym do projektu. shadcn/ui byłoby nadmiarowe i wymagałoby nadpisywania styli. Migracja z klas CSS do React+cva odbywa się stopniowo.
class-variance-authority (cva) Biblioteka do tworzenia typowanych wariantów komponentów React — definiujesz raz warianty (variant, size, intent) i dostajesz w pełni typowaną funkcję generującą klasy Tailwind. Integruje się z clsx i tailwind-merge dla obsługi warunkowych i nadpisanych klas. Pozwala stopniowo migrować komponenty CSS z phoenix-design-system do natywnych komponentów React.
Bez cva każdy komponent Button/Badge wymaga ręcznego zarządzania klasami warunkowymi — chaos przy wielu wariantach. cva daje type safety: TypeScript wie jakie warianty istnieją i pilnuje poprawności props.
TanStack Query v5 Biblioteka do zarządzania stanem serwera w React — fetching, caching, synchronizacja i aktualizacja danych z API. Automatycznie obsługuje loading states, error handling, background refetching i optimistic updates, eliminując setki linii boilerplate kodu. TanStack Query v5 wprowadza ulepszony infinite scrolling, streamlined API i lepszy TypeScript inference.
W CRM niemal każdy widok to dane z serwera. Bez TanStack Query każdy komponent potrzebuje własnego useEffect + useState + loading + error. TanStack zastępuje to kilkoma liniami z automatycznym cache invalidation.
React Hook Form + Zod React Hook Form to biblioteka formularzy minimalizująca re-rendery (uncontrolled components), Zod to TypeScript-first schema validation z pełnym type inference. Razem tworzą potężny duet — Zod definiuje schemat danych, Hook Form obsługuje formularz, walidacja dzieje się automatycznie z precyzyjnymi komunikatami błędów. Jeden schemat Zod może być reużyty zarówno na frontendzie jak i w API.
CRM ma dziesiątki formularzy (lead, klient, projekt, ticket, oferta...). Formik jest wolniejszy, własna walidacja to duplikacja logiki backendowej. Jeden Zod schema obsługuje frontend i może być współdzielony z backendem.
dnd-kit Nowoczesna, lekka biblioteka drag & drop dla React z pełnym wsparciem dla TypeScript i dostępności (keyboard DnD). Modułowa architektura pozwala budować pionowe listy, siatki i kanban boards bez zbędnego kodu. Obsługuje touch events, sensory (mouse/touch/keyboard) i smooth animations przez CSS transforms bez layout thrashing.
React Beautiful DnD nie jest już aktywnie rozwijany i ma problemy z React 18+. dnd-kit to jego następca — lżejszy, szybszy. Kluczowy dla Kanban leadów i ticketów.
TipTap v2 Headless rich text editor oparty o ProseMirror, w pełni kontrolowalny przez React i TypeScript, bez narzuconych styli — aplikujesz klasy phoenix-design-system bezpośrednio. Obsługuje tabele, listy, obrazy, mentions (@user), linki i można go rozszerzyć o custom nodes (np. bloki AI suggestions). Zapisuje treść jako JSON lub HTML, co ułatwia eksport do PDF przez Browsershot.
TinyMCE wymaga API key, ma narzucone style trudne do nadpisania i jest ciężki. TipTap jest headless (MIT) — stylujemy go phoenix-design-system bez walki z biblioteką. JSON output to lepszy format niż HTML dla wersjonowania dokumentów.
echarts + echarts-for-react Apache ECharts to jedna z najpotężniejszych bibliotek wykresów — obsługuje linie, słupki, pie, scatter, heatmapy, gauge, tree charts, mapy i wiele więcej. echarts-for-react to oficjalny wrapper React. Wykresy są w pełni konfigurowalane przez opcje JSON i responsywne out of the box.
Moduł SEO wymaga heatmap (godziny aktywności), gauge (score audytu), tree chart (struktura strony) — Recharts tego nie ma. ECharts zastępuje Recharts i Tremor jednocześnie, z większymi możliwościami i spójnym API.
React Router v7 Standard routingu dla React SPA z pełnym wsparciem dla nested routes, lazy loading (code splitting per route) i data loading patterns. React Router v7 unifikuje się z Remix oferując loaders i actions. Code splitting zapewnia że klient pobiera tylko kod potrzebny dla aktualnego widoku.
De facto standard — każdy React developer go zna, co ułatwia onboarding. Code splitting krytyczny przy dużej aplikacji z dziesiątkami widoków i modułów.
Zustand Minimalistyczna biblioteka do zarządzania globalnym stanem aplikacji (~1KB gzip), znacznie prostsza niż Redux bez boilerplate. Używana dla stanu który nie należy do serwera (TanStack Query) — aktywne powiadomienia, preferencje UI, stan sidebara, dane zalogowanego użytkownika. Nie wymaga Provider wrappera i działa poza komponentami React.
Redux to overhead dla stanu UI. React Context re-renderuje cały subtree. Zustand subskrybuje tylko do konkretnych wartości — idealne dla powiadomień real-time z Reverb.
Laravel Echo JavaScript biblioteka ułatwiająca subskrypcję kanałów WebSocket w Laravel, współpracująca natywnie z Reverb. Abstrahuje niskopoziomowe WebSocket API — prosty interfejs do nasłuchiwania eventów na prywatnych i publicznych kanałach. Używana do real-time powiadomień in-app, live aktualizacji ticketów i Kanban.
Pisanie własnego WebSocket klienta dla Laravel Broadcasting to niepotrzebna praca. Echo obsługuje automatycznie autoryzację prywatnych kanałów i reconnection logic.
react-select Zaawansowany komponent select z wbudowanym multi-select, autocomplete, async loading opcji i pełną customizacją. Zastępuje natywny HTML select i Choices.js z ClientOps Bootstrap. Styl nadpisywalny przez classNames API — kompatybilny z phoenix-design-system.
Natywny select HTML nie obsługuje multi-select z wyszukiwaniem. react-select jest standardem dla tego wzorca w React — używany m.in. w formularzu leadów (tagi) i przypisywaniu użytkowników do projektów.
react-dropzone Hook i komponent do obsługi drag & drop uploadu plików z walidacją typów, rozmiarów i podglądem. Integruje się z Spatie Media Library po stronie backendu. Używany w module dokumentów (załączniki) i profilu klienta.
Natywny input file nie obsługuje drag & drop. react-dropzone to lekki, elastyczny hook — łatwy do ostylowania komponentami phoenix-design-system.
react-day-picker Lekki, elastyczny date picker i date range picker dla React, bez zewnętrznych zależności. Styl w pełni kontrolowany przez CSS classes — kompatybilny z Tailwind. Używany w formularzach leadów (follow-up date), projektów (deadline) i filtrach tabeli.
flatpickr z ClientOps Bootstrap nie integruje się z React Hook Form. react-day-picker jest headless jak TipTap — aplikujemy klasy phoenix-design-system bezpośrednio.
@fullcalendar/react Pełnofunkcjonalny kalendarz eventów z widokami miesiąc/tydzień/dzień/lista i drag & drop eventów. Oficjalny wrapper React z TypeScript support. Używany w module projektów (harmonogram zadań) i CRM (follow-upy, spotkania).
Bezpośredni odpowiednik fullcalendar z ClientOps Bootstrap — ta sama biblioteka, oficjalny wrapper React, zero nauki od zera. Potrzebny dla widoku kalendarza CRM i harmonogramu projektów.
lucide-react Biblioteka ikon SVG jako komponenty React — następca Feather Icons, przepisany w TypeScript. Ponad 1000 ikon w spójnym stylu outline, każda jako osobny named export (tree-shakeable). Używana w całym interfejsie.
feather-icons i FontAwesome wymagają osobnych wrapperów. lucide-react jest natywnie dla React, tree-shakeable (zero nieużywanych ikon w bundle) i ma aktywny rozwój.
dayjs + lodash-es dayjs to lekka (~2KB) biblioteka do manipulacji datami (format, diff, parse, relative time) — bezpośredni zamiennik moment.js. lodash-es to tree-shakeable wersja lodash z utility funkcjami (debounce, throttle, groupBy, pick).
Obie biblioteki były używane w ClientOps Bootstrap i działają w React bez żadnych zmian. lodash-es zamiast lodash — tree-shaking eliminuje nieużywane funkcje z bundle.
Dlaczego React SPA zamiast Inertia.js? Panel klienta na subdomenach ({slug}.clientops.pl) wymaga oddzielnego frontendu, co oznacza konieczność budowania API. Skoro API jest i tak potrzebne, React SPA jest naturalnym wyborem — oba frontendy (admin + client) mogą współdzielić kod i korzystać z tego samego API.

Identyfikacja tenanta

Middleware wyciąga slug z subdomeny i wstrzykuje klienta do IoC container:

// app/Http/Middleware/IdentifyTenant.php
public function handle(Request $request, Closure $next)
{
    $host = $request->getHost();         // acme.clientops.pl
    $slug = explode('.', $host)[0];    // acme

    if ($slug === 'app' || $slug === 'api') {
        return $next($request);            // admin / api — skip
    }

    $tenant = Client::where('slug', $slug)->firstOrFail();
    app()->instance(Client::class, $tenant);
    app()->instance('tenant', $tenant);

    return $next($request);
}

Multi-tenancy — model danych

Wybór: Single Database z Global Scopes Dla tego projektu wystarczy jedna baza danych z client_id na każdej tabeli oraz automatycznym Global Scope. Prostsze w utrzymaniu i wystarczające na start. DB per tenant to nadmiarowa złożoność bez wymiernych korzyści na tym etapie. 
// Trait dla wszystkich modeli powiązanych z klientem
trait BelongsToTenant
{
    protected static function booted(): void
    {
        static::addGlobalScope(new TenantScope);

        static::creating(function ($model) {
            $model->client_id ??= app('tenant')?->id;
        });
    }
}

SLA breach monitoring

// Scheduled job — co 15 minut
class CheckSlaBreachesJob
{
    public function handle(): void
    {
        $breaching = Ticket::query()
            ->whereNull('first_responded_at')
            ->where('sla_response_due_at', '<', now()->addMinutes(30))
            ->get();

        $breaching->each(fn($ticket) =>
            $ticket->assignees->each->notify(new SlaBreachWarningNotification($ticket))
        );
    }
}

Generowanie PDF — kod kolejki

class GenerateDocumentPdfJob implements ShouldQueue
{
    public function handle(Browsershot $browsershot): void
    {
        $html = view('pdf.document', [
            'document' => $this->document,
            'client'   => $this->document->client,
        ])->render();

        $path = "documents/{$this->document->id}/v{$this->document->version}.pdf";

        $browsershot->html($html)
            ->format('A4')
            ->margins(20, 20, 20, 20)
            ->save(storage_path("app/{$path}"));

        $this->document->update(['pdf_path' => $path]);

        $this->document->creator->notify(new PdfReadyNotification($this->document));
    }
}

Zunifikowany inbox — System 2: wiadomości wewnętrzne

Oprócz notyfikacji systemowych (SLA, @mention, ticket) użytkownicy mogą wysyłać wiadomości do innych adminów lub do klientów bezpośrednio w CRM — bez wychodzenia poza system. Oba strumienie trafiają do jednego inboxu.

Notyfikacje systemowe

Wiadomości admin → admin

Wiadomości admin → klient

Security
Role i uprawnienia

System uprawnień oparty o Spatie Laravel Permission (RBAC). Sześć ról z hierarchią dostępu — dwie po stronie admina, cztery po stronie klienta.

Hierarchia ról

  • super-admin
    Dostęp do wszystkiego. Zarządzanie ustawieniami systemu, konfiguracją subdomen, planami klientów, integracjami API (Google, AI). Tylko właściciel systemu.
  • admin
    Pełny dostęp do CRM — leady, klienci, projekty, SEO audyty, dokumenty. Zarządzanie zespołem, przeglądanie wszystkich ticketów, konfiguracja SLA.
  • manager
    Zarządza przypisanymi projektami i klientami. Może tworzyć dokumenty i zadania, obsługiwać tickety. Bez dostępu do ustawień systemu i finansów.
  • employee
    Widzi tylko przypisane projekty i zadania. Może komentować tickety i tworzyć dokumenty w ramach swoich projektów. Brak dostępu do danych innych klientów.
  • client-admin
    (Panel klienta) Pełny dostęp do panelu swojego klienta. Widzi projekty, SEO raporty, dokumenty, może zarządzać użytkownikami swojej organizacji i zgłaszać tickety.
  • client-user
    (Panel klienta) Ograniczony dostęp — tylko podgląd przypisanych projektów i dokumentów, zgłaszanie ticketów. Bez dostępu do raportów finansowych i SEO.

Izolacja danych między klientami

Global Scope jako pierwsza linia obrony Każdy model z trait BelongsToTenant automatycznie filtruje dane po client_id. Dodatkowo Gate policies sprawdzają uprawnienia na poziomie pojedynczego zasobu. Dwupoziomowa ochrona: scope na poziomie modelu + policy na poziomie kontrolera. 
// Przykład Policy
class ProjectPolicy
{
    public function view(User $user, Project $project): bool
    {
        // Global scope już odfiltrował inne klientów
        // Tu sprawdzamy czy user ma dostęp do tego projektu
        return $user->hasRole(['admin', 'manager'])
            || $project->assignees->contains($user);
    }
}
