Stałe procesy działające w tle

ClientOps nie jest systemem pasywnym — kilka procesów działa nieprzerwanie, niezależnie od aktywności użytkowników:

Proces Częstotliwość Komponent Obciążenie
CheckSlaBreachesJob co 5 minut Laravel Scheduler → Queue Zapytanie DB — skanuje aktywne tickety i projekty
Analiza SEO — pipeline 1×/tydz. per projekt Horizon (5 kolejek) I/O sieciowy, Google APIs, AI API, DB write
Miesięczny audyt + PDF 1×/miesiąc per klient Horizon + Browsershot Headless Chromium — ~1 GB RAM / instancja
WebSocket (Reverb) ciągłe połączenia Laravel Reverb + Redis Długotrwałe połączenia TCP, pub/sub Redis

TimescaleDB — konfiguracja dla 5 000 klientów

Tabela seo_keyword_ranks będzie rosnąć szybko: 15 000 projektów × średnio 50 słów kluczowych × 4 snapshoty/miesiąc = 3 000 000 rekordów/miesiąc. TimescaleDB jest tu niezbędny.

-- Hypertable z partycjonowaniem miesięcznym
SELECT create_hypertable('seo_keyword_ranks', 'checked_at',
  chunk_time_interval => INTERVAL '1 month'
);

-- Polityka retencji: zachowaj 2 lata
SELECT add_retention_policy('seo_keyword_ranks', INTERVAL '2 years');

-- Continuous aggregate — tygodniowe trendy (dla wykresów echarts)
CREATE MATERIALIZED VIEW seo_rank_weekly
WITH (timescaledb.continuous) AS
  SELECT time_bucket('1 week', checked_at) AS week,
         keyword_id,
         project_id,
         AVG(position)    AS avg_position,
         MIN(position)    AS best_position,
         SUM(impressions) AS total_impressions
  FROM seo_keyword_ranks
  GROUP BY 1, 2, 3;

-- Polityka odświeżania agregatu
SELECT add_continuous_aggregate_policy('seo_rank_weekly',
  start_offset => INTERVAL '3 weeks',
  end_offset   => INTERVAL '1 hour',
  schedule_interval => INTERVAL '1 day'
);

Optymalizacja
Krytyczne Indeksy Bazy Danych

Bez odpowiednich indeksów system przy 5 000 klientach przestanie odpowiadać. Każde zapytanie w multi-tenant systemie zaczyna od filtrowania po client_id — to musi być zawsze zindeksowane.

Indeksy SLA — krytyczne dla CheckSlaBreachesJob

Job uruchamiany co 5 minut musi skanować tylko aktywne rekordy. Partial index dramatycznie zmniejsza rozmiar skanowanego zestawu.

-- Tylko aktywne tickety z przekroczonym terminem SLA
CREATE INDEX idx_tickets_sla_response
  ON tickets (sla_response_due_at)
  WHERE status NOT IN ('resolved', 'closed')
    AND sla_response_due_at IS NOT NULL;

CREATE INDEX idx_tickets_sla_resolution
  ON tickets (sla_resolution_due_at)
  WHERE status NOT IN ('resolved', 'closed')
    AND sla_resolution_due_at IS NOT NULL;

-- Nienotyfikowane naruszenia SLA
CREATE INDEX idx_sla_incidents_pending
  ON sla_incidents (breached_at)
  WHERE notified = false;

Indeksy Multi-tenancy — Global Scopes

Trait BelongsToTenant dodaje WHERE client_id = ? do każdego zapytania. Bez composite index każde zapytanie to full table scan.

-- Composite indeksy per tabela — najczęstsze kombinacje zapytań
CREATE INDEX idx_tickets_client_status      ON tickets (client_id, status);
CREATE INDEX idx_projects_client            ON projects (client_id);
CREATE INDEX idx_leads_client_status        ON leads (client_id, status);
CREATE INDEX idx_documents_documentable     ON documents (documentable_type, documentable_id);
CREATE INDEX idx_seo_analyses_project       ON seo_analyses (project_id, created_at DESC);
CREATE INDEX idx_audits_client_type         ON audits (client_id, type, created_at DESC);

-- Unikalność slugów w scope
CREATE UNIQUE INDEX idx_projects_slug_client    ON projects (client_id, slug);
CREATE UNIQUE INDEX idx_leads_slug_client       ON leads (client_id, slug);
CREATE UNIQUE INDEX idx_documents_slug          ON documents (documentable_type, documentable_id, slug);

-- Dla zapytań o historię pozycji konkretnego słowa kluczowego

Wąskie Gardła i Strategie Optymalizacji
📄
PDF Generation — Browsershot / Chromium
Najcięższy pojedynczy job w systemie
Jeden Chromium zużywa ~1 GB RAM i 30–90 sekund CPU. Przy 500 PDF/dzień (5k klientów) kolejka pdf musi pracować non-stop przez 4–12 godzin.
Dedykowana kolejka pdf z limitem workerów = RAM / 1.2 GB. Planowanie generowania w oknach nocnych (00:00–06:00) przez Laravel Scheduler. Priorytetyzacja: audyty triggered ręcznie > automatyczne miesięczne.
Alternatatywa: wFPDF/DomPDF dla prostych raportów (szybsze), Browsershot tylko dla skomplikowanych layoutów z wykresami echarts.
⏰
SLA Monitoring — CheckSlaBreachesJob co 5 minut
Skalowanie zapytań przy dużej liczbie rekordów
Przy 5 000 klientach tabela tickets może mieć 100 000+ rekordów. Full table scan co 5 minut = pożeranie CPU i I/O bazy danych.
Partial index na aktywnych ticketach (patrz sekcja 5). Redis cache dla ostatnio sprawdzonych rekordów — skip jeśli sprawdzono < 5 min temu. Zamiast skanować całą tabelę: WHERE sla_response_due_at < NOW() AND notified = false.
Docelowo: zamiast pollingu co 5 min — event-driven (Carbon + Redis Sorted Set: ticket → score = timestamp breach). Job sprawdza tylko top N z ZRANGEBYSCORE.
🔑
Google APIs — Rate Limiting i Quota
GSC API: 200 req/100s per user token; GA4: 10 concurrent requests
Przy 15 000 projektach i 4 analizach/miesiąc = 60 000 wywołań GSC/miesiąc. Przy zbyt wielu równoległych workerach — API zwróci 429 Too Many Requests dla całej aplikacji.
Kolejka google-apis z throttle middleware opartym na Redis: max 20 jobów GSC/minutę per OAuth token. Exponential backoff przy 429. Rozłożenie jobów w czasie — nie planuj wszystkich analiz SEO na ten sam dzień tygodnia.
Każdy klient autoryzuje własny Google OAuth token — przy 100+ klientach limity dotyczą per token, nie globalnie. To znacznie zwiększa przepustowość.
🤖
AI Processing — Koszt i Latencja API
OpenAI / Claude — każda analiza semantyczna to koszt i czas
Analiza semantyczna jednej strony przez Claude/OpenAI: ~0.5–2 USD + 5–30 sekund latencji. Przy 15 000 projektach × 4/miesiąc = koszt może przekroczyć $30 000/miesiąc bez optymalizacji.
Cache wyników analizy semantycznej w Redis z TTL 7 dni — jeśli treść strony nie zmieniła się (hash treści), nie wysyłaj ponownie do AI. Batch processing: kilka stron w jednym promptcie tam gdzie możliwe. Limit analizowanych stron per projekt (np. top 10 stron wg ruchu z GA4).
Rozważyć modele: Claude Haiku lub GPT-4o-mini dla masowych analiz (tańsze 10×), Claude Opus/GPT-4 tylko dla finalnego raportu.
🌐
Crawler — Ethical Crawling i Timeouty
Crawlowanie cudzych stron musi szanować robots.txt i nie przeciążać serwerów klientów
CrawlWebsiteJob może trwać 10–300 sekund zależnie od rozmiaru strony. Jednoczesne crawlowanie 8+ stron = duże zużycie sieci i CPU. Brak respektu dla robots.txt lub zbyt agresywne crawlowanie może zablokować IP serwera.
Crawler workers na dedykowanych maszynach (osobne IP lub IP rotation). Respektowanie robots.txt i Crawl-delay. Limit: max 1 request/sekunda per domena. Timeout 300s z graceful cancellation. Cache crawl results na 24h — ponowne uruchomienie analizy nie crawluje od nowa.
Alternatywa dla własnego crawlera: zewnętrzne API (Screaming Frog API, Ahrefs API, SEMrush API) — drogie, ale eliminuje całkowicie problem crawlowania po stronie serwera.
