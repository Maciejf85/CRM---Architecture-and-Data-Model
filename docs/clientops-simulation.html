Pierwsza aktywność — logowanie rozmowy telefonicznej

tabela: lead_activities

type:           "call"  -- enum: call|email|meeting|note|status_change

                 Strona: agencja-xyz.pl. Chce zobaczyć audyt."
occurred_at:   "2026-03-29 10:15:00"

Status leada zmienia się na contacted. Event LeadStatusChanged jest wyrzucany — listener automatycznie przesuwa kartę Kanbana do odpowiedniej kolumny.

event → listener: przesunięcie karty Kanban
// LeadStatusChanged event
LeadObserver::updated($lead)
  → event(new LeadStatusChanged($lead, 'new', 'contacted'))

// SyncLeadKanbanCardListener
KanbanCard::where('cardable_type', 'Lead')
           ->where('cardable_id', 1)
           ->update(['column_id' => 'contacted'])
ℹ
activity_log — każda akcja jest rejestrowana
Spatie Activity Log zapisuje każdą zmianę: kto, co, kiedy, diff. Wpis zawiera causer_id: 3 (Ania), subject_type: "Lead", properties.old.status: "new", properties.new.status: "contacted".

Krok 5 · Konwersja
Konwersja — lead staje się klientem, aktywacja subdomeny

Tomasz odpisuje — zgadza się. Admin klika Konwertuj lead. System wykonuje całą operację w jednej transakcji bazodanowej.

LeadConversionService::convert($lead) — DB::transaction
// 1. Tworzy klienta
clients::create([
  'slug'  => 'agencja-xyz',          // immutable slug
  'name'  => 'Agencja Kreatywna XYZ',
  'email' => 't.kowalski@agencja-xyz.pl',
])

// 2. Kopiuje kontakty
contacts::where('lead_id', 1)
        ->update(['client_id' => 1])

// 3. Zamyka lead jako won
leads::update([
  'status'       => 'won',
  'converted_at' => now(),
  'client_id'    => 1,
])

// 4. Aktualizuje ofertę
offers::update([
  'client_id'              => 1,
  'status'                 => 'accepted',
  'accepted_at'            => now(),
  'converted_to_client_at' => now(),
])

⚠
CheckSlaBreachesJob — co 5 minut w tle
Scheduler uruchamia job co 5 minut. Sprawdza wszystkie otwarte tickety — czy przekroczone sla_response_due_at lub sla_resolution_due_at. Jeśli tak — tworzy sla_incident i wysyła powiadomienie do managera. Tomasz jako klient NIE widzi incydentów SLA — to widok tylko admin/manager.

Autoryzacja — dwa poziomy

Poziom Mechanizm Przykład
Rola Spatie Permission hasRole('manager') — czy w ogóle może zarządzać projektami
Rekord Laravel Policy ProjectPolicy::update() — czy jest przypisany do tego konkretnego projektu
Tenant Global Scope Automatyczny filtr WHERE client_id = 1 — nie zobaczy danych innych klientów

Krok 8 · Automatyzacja
Miesięczny audyt automatyczny — pipeline i powiadomienia

Pierwszego dnia miesiąca o 6:00 scheduler Laravel uruchamia GenerateMonthlyAuditJob. Tym razem analiza SEO jest składową audytu (nie samodzielną) — wyniki są agregowane w raporcie zbiorczym.

Scheduler — cron: 0 6 1 * *
// app/Console/Kernel.php
$schedule->job(new GenerateMonthlyAuditJob)
          ->cron('0 6 1 * *')
          ->withoutOverlapping();
tabela: audits + seo_analyses — audyt ze składową
-- nadrzędny audyt
audits:
  id: 1, client_id: 1, project_id: 1
  type: "seo", status: "completed"
  triggered_by: "scheduler"

-- analiza jako składowa audytu (audit_id ustawiony!)
seo_analyses:
  id: 3, audit_id: 1    ← różnica vs analiza samodzielna
  project_id: 1, client_id: 1
  status: "completed"
