Kluczowa zasada architektury Lead → Klient → Projekt to trzy osobne stany relacji biznesowej. Lead staje się klientem przez eksplicytną konwersję (event + listener). Projekt jest tworzony dla klienta — nigdy bezpośrednio dla leadu. Nie skracaj tej ścieżki.

Multi-tenancy — dwa poziomy izolacji

Poziom 1 — Organizacja (top-level SaaS tenant): każda tabela biznesowa zawiera organization_id. Trait BelongsToOrganization dodaje Global Scope automatycznie. Middleware IdentifyOrganization identyfikuje organizację z sesji zalogowanego usera.

// IdentifyOrganization — identyfikacja z sesji usera (panel admina)
$org = Organization::find($request->user()->organization_id);
app()->instance('organization', $org);

// Trait BelongsToOrganization — Global Scope + auto-fill organization_id
static::addGlobalScope('organization', fn($q) => $q->where('organization_id', app('organization')->id));
static::creating(fn($m) => $m->organization_id = app('organization')->id);

Poziom 2 — Klient (panel klienta na subdomenach): middleware IdentifyTenant identyfikuje klienta z subdomeny. Obsługuje dwa formaty URL:

// 3-częściowy (backward compat): acme.clientops.pl
$tenant = Client::withoutOrganizationScope()->where('slug', $parts[0])->first();

// 4-częściowy (z org): acme.kordit.clientops.pl
$org    = Organization::where('slug', $parts[1])->first();
$tenant = Client::withoutOrganizationScope()
    ->where('slug', $parts[0])
    ->where('organization_id', $org->id)->first();

Organizations — top-level SaaS tenant Jeden CRM może obsługiwać wiele niezależnych organizacji (agencji, klientów SaaS). Każda organizacja ma własną izolowaną przestrzeń danych: klientów, leady, projekty, dokumenty, tickety. Użytkownicy są przypisani do jednej organizacji przez organization_id w tabeli users.

Polityka slug w systemie Slug generowany automatycznie przy tworzeniu rekordu (spatie/laravel-sluggable, onUpdate = false). Zmiana tytułu/nazwy nie aktualizuje slugu. Ręczna zmiana slugu przez admina zapisuje stary slug do slug_redirects — middleware zwraca 301 Redirect na aktualny URL. Kolizja slugu w tym samym scope → auto-suffix: umowa → umowa-2 → umowa-3.

Event-driven — konwersja leadu

// Event: LeadWonEvent::class (fired gdy leads.status → 'won')
class LeadWonListener
{
    public function handle(LeadWonEvent $event): void
    {
        $lead   = $event->lead;
        $client = Client::create([
            'name'                  => $lead->company_name,
            'slug'                  => Str::slug($lead->company_name),
            'converted_from_lead_id'=> $lead->id,
            'sla_policy_id'         => SlaPolicy::default()->id,
        ]);

        $lead->contacts->each(fn($c) => $c->update(['client_id' => $client->id]));
        $lead->update(['converted_to_client_id' => $client->id, 'converted_at' => now()]);

        activity('crm')
            ->on($lead)
            ->log('converted_to_client');
    }
}

Scheduled job — CheckSlaBreachesJob (co 5 min)

Schedule::job(CheckSlaBreachesJob::class)->everyFiveMinutes();

CheckSlaBreachesJob skanuje:
├── tickets: sla_response_due_at < now() AND first_responded_at IS NULL
│     → INSERT sla_incidents (breach_type: response)
│     → Notification do admina (mail + SMS jeśli priority: critical)
│     → Kanban karta → kolor czerwony
│
├── tickets: sla_resolution_due_at < now() AND resolved_at IS NULL
│     → INSERT sla_incidents (breach_type: resolution)
│     → Notification do managera
│
└── projects: deadline < now() AND status != 'completed'
      → INSERT sla_incidents (breach_type: deadline)
      → Notification in-app + email
Jeden ticket może mieć wiele sla_incidents Np. najpierw response_breach po 4h bez odpowiedzi, potem resolution_breach po 24h. Każdy incydent to osobny rekord z własnym detected_at i niezależnym procesem acknowledged.

AuditRequestedEvent
  │
  ├── CrawlWebsiteJob           (queue: crawlers)
  │     → seo_audit_pages       — wyniki per URL (title, H1, meta, treść, linki)
  │     → seo_issues            — broken links, missing meta, duplicate content

  │     → seo_monthly_snapshots — organic_sessions, conversions, avg_ctr, avg_position

  ├── FetchPageSpeedJob         (queue: google-apis) ← równolegle
  │     → seo_cwv_snapshots     — LCP, CLS, INP, FCP, TTFB per URL (hypertable)
  │     → audit_scores          — kategoria: performance (score 0-100)
  │     → audit_issues          — kategoria: performance (lcp_fail, cls_fail...)
  │
  ├── FetchBacklinksJob         (queue: external-apis)
  │     → seo_backlinks         — referring_domain, DR, anchor, first/last_seen
  │     → seo_monthly_snapshots — new_backlinks, lost_backlinks, referring_domains
