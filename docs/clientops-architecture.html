Podział systemu na warstwy

System podzielony na trzy warstwy logiczne. Każda warstwa obsługuje inny obszar biznesowy i zależy od warstwy poprzedniej — nie można mieć projektu bez klienta, nie można mieć klienta bez kontaktu.

Minimalny zestaw tabel na start

Kolejność wdrożenia odpowiada warstwom — nie można tworzyć tabeli wyższej warstwy bez fundament niższej.

Kluczowa zasada architektury Nie łącz wszystkiego w jeden byt. Lead, klient i projekt to trzy oddzielne stany relacji biznesowej. Lead staje się klientem przez konwersję — projekt jest tworzony dla klienta, nigdy dla leadu.

Krok "Oferta" — nowy element względem spec v1.0.0 Oferta to osobny byt między leadem a klientem. Lead może mieć wiele wersji oferty (renegocjacje). Akceptacja oferty wyzwala konwersję leadu na klienta i opcjonalne automatyczne utworzenie projektu.

Dwa osobne byty o różnym źródle pochodzenia i celu. Powiązane — ale niezależne.

Scheduled job — monitoring co 5 minut

// app/Console/Commands/CheckSlaBreaches.php
Schedule::job(CheckSlaBreachesJob::class)->everyFiveMinutes();

CheckSlaBreachesJob skanuje:
├── tickets: sla_response_due_at < now() AND first_responded_at IS NULL
│     → nowy sla_incident (response) + SMS do admina (jeśli critical)
│     → ticket na Kanban → kolor czerwony
│
├── tickets: sla_resolution_due_at < now() AND resolved_at IS NULL
│     → nowy sla_incident (resolution) + email do managera
│
└── projects: deadline < now() AND status != 'completed'
      → nowy sla_incident (deadline) + powiadomienie in-app
Jeden ticket może mieć wiele sla_incidents Np. najpierw response_breach (nie odpowiedziano w czasie), potem resolution_breach (nie rozwiązano w czasie). Każdy incydent jest osobnym rekordem z własnym detected_at.

Audyty i analizy SEO — relacja

Analiza SEO może być bytem samodzielnym (uruchamianym ad hoc) lub składową audytu (grupowaną razem z innymi typami analiz). Obie ścieżki są obsługiwane przez jeden model z opcjonalnym audit_id.

Dwa tryby użycia

Analiza SEO — samodzielna
Admin uruchamia analizę dla projektu bez tworzenia audytu. seo_analyses.audit_id = null. Widoczna w sekcji "Analizy SEO" projektu. Typowy przypadek: miesięczny monitoring pozycji.
Analiza SEO — składowa audytu
Audit grupuje wiele analiz i typów (SEO + techniczne + UX). seo_analyses.audit_id = 42. Wyniki agregowane w raporcie audytu z jednym scoringiem. Typowy przypadek: kompleksowy audyt kwartalny.

Schemat tabel

-- Byt nadrzędny — grupuje analizy różnych typów
audits
  - project_id
  - type: seo | technical | ux | security | full
  - status: pending | running | completed | failed
  - overall_score     -- 0-100, agregat wszystkich składowych
  - started_at, completed_at
  - created_by

-- Analiza SEO — samodzielna LUB składowa audytu
seo_analyses
  - project_id
  - audit_id (nullable)  -- null = samodzielna, int = część audytu
  - status: pending | running | completed | failed
  - score               -- 0-100
  - keywords_count
  - issues_count
  - gsc_connected: bool -- czy dane z Google Search Console
  - ga4_connected: bool -- czy dane z Google Analytics 4
  - completed_at

-- Błędy wykryte w analizie
seo_issues
  - seo_analysis_id
  - type: missing_meta | broken_link | duplicate_content | ...
  - severity: critical | warning | info
  - url
  - description
  - fixed_at (nullable)

Dlaczego nie jeden model "seo_audits"? Rozdzielenie audits od seo_analyses pozwala w przyszłości dodać inne typy analiz (UX, security, performance) bez zmiany struktury. Audyt staje się "folderem" — analiza SEO to jeden z dokumentów w tym folderze.
